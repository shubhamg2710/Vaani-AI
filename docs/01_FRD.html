Executive Summary

This document defines the complete set of functional and non-functional requirements for Vaani AI — a real-time multilingual conversational AI platform targeting B2B customers (Startups, SMBs, and Enterprise) across the BFSI, Automotive, Government, and Healthcare domains. The platform enables businesses to deploy production-grade voice AI agents capable of handling inbound and outbound calls in the customer's native Indian or global language, with zero hallucination guarantees and a self-improving feedback loop.

The MVP delivers three core capabilities: (1) a Generalized Real-Time Conversational Agent using a Cascaded Speech-to-Speech (STS) pipeline — Speech → ASR → LLM → TTS → Speech; (2) an Industry-Specific BFSI Agent covering banking, insurance, and loans; and (3) an Industry-Specific Automotive Agent covering service booking, sales support, and roadside assistance (RSA). Healthcare and Government verticals are planned for Phase 2.

All requirements in this document are written to be implementation-ready. Each functional requirement includes a description, detailed sub-requirements, and a structured acceptance criteria table with measurable pass/fail conditions suitable for QA engineering and UAT sign-off.

Product Vision

"Enable any business to deploy a voice AI agent that speaks every customer's language, resolves queries instantly, and gets smarter with every conversation."

Vaani AI is built on the belief that language should never be a barrier to accessing financial services, vehicle support, or government information. By combining state-of-the-art multilingual ASR, LLM-powered reasoning, and native TTS — all in a latency-optimized real-time pipeline — Vaani AI makes enterprise-grade voice AI accessible to any business in India and beyond.

Key differentiators:

  • 20+ languages including 10+ Indic languages (Hindi, Tamil, Telugu, Marathi, Bengali, Kannada, Gujarati, Punjabi, Malayalam, Odia)
  • Code-switching support: Hinglish, Tanglish, Bamglish — no re-initialization, single unified model
  • <500ms end-to-end response latency (ASR + LLM + TTS combined at P95)
  • Anti-hallucination architecture: RAG-only factual answers, confidence thresholds, guardrail layer
  • Self-improving via call outcome learning (CSAT, escalation, resolution signals)
  • Telephony-optimized: 8kHz noisy audio, G.711 codec, background noise handling
  • Full B2B multi-tenant platform with white-labeling, analytics, and CRM integrations

Stakeholders & User Personas

1 End User / Caller

Indian consumer calling a bank, car dealership, or government helpline. Varied tech literacy. May code-switch between languages. May be impatient, distracted, or calling from a noisy environment. Primary driver of CSAT and resolution metrics.

2 Business Admin

Operations manager at HDFC Bank, Maruti Suzuki dealership, or an SMB configuring the agent. Non-technical. Uses the no-code dashboard for script building, KB management, and analytics. Needs simple UX with no engineering dependency.

3 ML Ops Engineer

Fine-tunes ASR and LLM models, monitors WER and intent accuracy, manages training pipelines, and executes canary deployments. Works with Label Studio for human annotation and Weights & Biases for experiment tracking.

4 Developer / Integrator

Connects Vaani AI APIs to existing CRM (Salesforce), core banking systems (Finacle, Flexcube), and dealer management systems. Uses REST and WebSocket APIs. Needs reliable documentation, sandbox environments, and pre-built integration adapters.

5 Compliance Officer

Ensures adherence to RBI guidelines, IRDAI regulations, GDPR, and India's DPDP Act 2023. Reviews audit logs, verifies PII masking, data localization, call recording retention policies, and signs off on data erasure requests.

F1 Real-Time Multilingual ASR with Streaming P0 Critical

The system shall convert spoken audio to text in real-time using a streaming pipeline that returns partial transcripts as speech occurs. The ASR must support a minimum of 20 languages including all major Indic languages, and be optimized for telephony-quality audio.

Detailed Requirements
  • Stream audio in chunks of 100–200ms via WebSocket connection
  • Return partial (interim) transcripts within 300ms of speech onset
  • Return final transcript within 500ms of end-of-utterance detection
  • Support 8kHz telephony audio (G.711 mu-law/A-law codec) AND 16kHz web audio (PCM/WAV/WebM)
  • Supported languages: Hindi (hi-IN), Tamil (ta-IN), Telugu (te-IN), Marathi (mr-IN), Bengali (bn-IN), Kannada (kn-IN), Gujarati (gu-IN), Punjabi (pa-IN), Malayalam (ml-IN), Odia (or-IN), English US (en-US), English UK (en-GB), English India (en-IN), German (de-DE), French (fr-FR), Spanish (es-ES), Italian (it-IT), Portuguese (pt-BR), Mandarin Chinese (zh-CN), Arabic (ar-SA)
  • Handle background noise, music-on-hold, telephony compression artifacts, and channel bandwidth limitations
  • WER targets (clean audio): ≤8% for Hindi and English; ≤12% for other Indic languages
  • Support speaker diarization to distinguish user speech from agent speech in recordings
  • Apply Voice Activity Detection (VAD) to avoid processing silence/noise as speech
  • Implement RNNoise or equivalent pre-processing for noise suppression at 8kHz

F2 Language Detection & Code-Switch Handling P0 Critical

The system shall automatically detect the caller's language from the first few seconds of speech without requiring any user selection. The system must natively handle mid-sentence language switching (code-switching) without requiring model re-initialization or call interruption.

Detailed Requirements
  • Auto-detect language from first 2–3 seconds of natural speech with ≥90% accuracy
  • Support token-level language identification within a single mixed-language utterance
  • Handle Hinglish (Hindi-English), Tanglish (Tamil-English), Kanglish (Kannada-English), and Bamglish (Bengali-English) natively via a unified model
  • No model switching or call re-initialization required during code-switching events
  • Generate a language confidence score (0.0–1.0); flag utterances with confidence <0.75 for fallback handling
  • Caller may explicitly state language preference (e.g., "Please speak to me in Hindi"); system overrides auto-detection
  • Session-level language lock: once language confirmed, retain across all turns unless caller explicitly changes

F3 LLM Orchestration & Intent Resolution P0 Critical

The LLM orchestration layer shall understand user intent, extract entities, manage slot filling, maintain conversation state, and generate accurate, helpful, domain-grounded responses. All factual content must be sourced from the RAG pipeline. The LLM shall also be capable of triggering backend tool calls (APIs) to perform real actions such as balance lookups or service bookings.

Detailed Requirements
  • Intent classification: identify primary intent from each utterance (e.g., check_balance, report_fraud, book_service, request_loan_status)
  • Entity extraction: extract named entities — account number, vehicle registration, date/time, amount, OTP, PIN (masked), name, location
  • Slot filling: track which required parameters are missing for a given intent; generate targeted follow-up questions to collect them
  • Context retention: maintain full conversation context within a call for up to 50 dialogue turns
  • RAG integration: for every factual query, retrieve top-5 relevant chunks from vector DB before generating response
  • Streaming response: use LLM streaming API so first tokens are forwarded to TTS as soon as generated
  • Tool calling: invoke backend APIs (balance lookup, service booking, OTP dispatch, account status) via structured function calls
  • Latency target: first token from LLM within 200ms (P95) under normal load conditions
  • Conciseness: responses calibrated for voice — concise, structured, no markdown, no lists in audio

F4 Context & Memory Management P1 High

The system shall maintain short-term conversational memory within a call (intra-call context) and long-term caller memory across multiple calls (cross-call profile), subject to caller consent and applicable data protection regulations.

Short-Term Memory (Intra-Call)
  • Store full conversation transcript in Redis with a unique session ID for the call duration
  • Maintain a sliding context window of the last 20 dialogue turns for LLM inference (to manage token limits)
  • Track resolved and pending intents within the call; never re-ask for confirmed information
  • Carry extracted entities across turns: account number confirmed in turn 2 reused in turn 7 without re-prompting
  • Session data cleared from Redis within 24 hours of call end
Long-Term Memory (Cross-Call)
  • Store caller profile in PostgreSQL: name, language preference, frequent query types, satisfaction history, last call summary
  • Inject cross-call context into system prompt at call start: "Previous call (DD-MM-YYYY): [summary]"
  • Caller can opt out of cross-call memory storage (GDPR/DPDP compliance); opt-out removes all stored data within 24 hours
  • Cross-call context lookup completes within 200ms of call connect
  • Last interaction summary generated automatically at call end by LLM summarization step

F5 Anti-Hallucination & Domain Boundary Enforcement P0 Critical

The agent must never fabricate product details, interest rates, regulatory information, or any factual claims. All factual responses must be grounded exclusively in verified knowledge base content retrieved via RAG. The system must also prevent the agent from answering questions outside its configured domain.

Detailed Requirements
  • All factual responses must be derived from RAG-retrieved knowledge base content only — LLM must not infer or fabricate numeric values
  • RAG retrieval confidence threshold: if top chunk similarity score <0.70, agent responds: "I don't have that specific information — let me connect you to a specialist."
  • Domain boundary check: queries outside the configured agent domain (e.g., a banking agent asked about cricket scores) must trigger a graceful out-of-domain response and redirect
  • Numeric facts (interest rates, processing fees, EMI amounts, deadlines) must ONLY be sourced from verified KB entries — never inferred by LLM generation
  • Post-generation guardrail layer: validates LLM output for hallucination signals (numeric contradictions, claims not found in retrieved context, out-of-domain factual assertions)
  • KB entries for rates and product information must be updated within 1 business day of any change by the business
  • Each response metadata includes the KB chunk IDs used for audit trail

F6 Distraction & Off-Topic Handling P1 High

The agent shall gracefully handle user distractions, irrelevant questions, and deliberate attempts to take the conversation off-topic, redirecting the conversation back to the pending task without being dismissive or rude. After two failed redirection attempts, the agent must offer human escalation.

Detailed Requirements
  • Maximum 2 off-topic redirect attempts before offering human agent escalation
  • Distraction categories: casual conversation, personal questions directed at the agent, competitor inquiries, unrelated complaints, testing/probing the AI identity
  • Each distraction category has a pre-approved, tenant-configurable graceful response template
  • After each redirect, agent always returns explicitly to the last pending task or unfilled slot
  • Agent identity questions handled specifically: "I'm Vaani AI's virtual assistant — I'm here to help with [domain]. Let me get back to [task]."
  • Feedback distractions acknowledged empathetically before redirect: "I understand your frustration — I'll note that. Now, let me help you with [pending task]."
  • All distraction handling events logged with category label for analytics review
Response Examples
  • User: "Are you a robot?" — Agent: "I'm Vaani AI's virtual assistant, here to help you with your banking queries. Now, about your account balance..."
  • User: "What's the weather today?" — Agent: "I specialize in banking queries. Let me help you finish what we were working on — your [pending task]."
  • User: "Your app is terrible" — Agent: "Thank you for that feedback — I've noted it. Now, let me make sure I resolve your query today..."

F7 TTS with Streaming & Language-Native Prosody P0 Critical

The system shall convert LLM text responses to natural, language-appropriate speech using streaming TTS that begins audio output on the first sentence fragment without waiting for the full response to complete. Each language must use a native voice model — not a translated or accented fallback.

Detailed Requirements
  • Streaming TTS: begin audio output on first complete sentence; do not wait for full LLM response before starting TTS
  • First audio chunk delivered within 75–150ms of TTS request (P95 target: ≤150ms)
  • Separate language-native voice model per language — no cross-language accent bleed
  • Voices available for: hi-IN, ta-IN, te-IN, mr-IN, bn-IN, kn-IN, gu-IN, pa-IN, en-IN, en-US, en-GB, and additional supported languages
  • SSML support: pauses (<break>), emphasis (<emphasis>), number pronunciation (₹50,000 → "fifty thousand rupees"), date and time formatting, proper noun pronunciation
  • Voice persona: configurable per tenant — name, gender (male/female/neutral), speech rate, pitch
  • White-label persona: HDFC Bank deploys as "Priya," Maruti Suzuki deploys as "Ravi"
  • Output format: 8kHz PCM for telephony delivery via Twilio/Exotel; 24kHz MP3 for web SDK

F8 Dialogue & Turn Management P0 Critical

The system shall manage real-time conversation turn-taking, including barge-in detection, end-of-utterance detection, silence handling, double-speak resolution, and maximum turn duration enforcement. These mechanisms must work reliably over telephony audio.

Detailed Requirements
  • Barge-in: detect when the user speaks while the agent TTS is playing; immediately stop TTS audio output and begin processing the new user input
  • End-of-utterance: detect speech completion using VAD silence threshold of 500ms (configurable per tenant from 300ms to 1500ms)
  • Silence handling: after 5 seconds of user silence, agent prompts "Are you still there?"; after 10 seconds total silence, agent offers to repeat or escalate
  • Double-speak: if user and agent audio overlap, agent audio is suppressed and user's speech is processed with priority
  • Max turn duration: configurable per script state — e.g., 30 seconds for simple intent confirmation, 120 seconds for complex documentation queries
  • Hold music: if backend API call takes >2 seconds, play configurable hold music/message to prevent caller disengagement
  • Interruption recovery: after barge-in and user turn processed, agent continues from logical next point (not from interrupted sentence start)

F9 Call Flow & Script Engine P0 Critical

The system shall execute configurable call scripts (conversation flows) defined by business admins via a no-code visual builder, with conditional branching, variable interpolation, slot tracking, and full versioning support. Pre-built industry templates shall accelerate BFSI and Automotive deployments.

Detailed Requirements

  • Industry templates: pre-built BFSI script (balance inquiry, loan status, fraud reporting, account opening) and Automotive script (service booking, vehicle status, RSA request)
  • Full script execution log per call: each state transition, turn, intent, and variable resolution stored with timestamps

F10 Knowledge Base Management (RAG) P1 High

The system shall maintain a domain-specific, searchable knowledge base per tenant, used to ground all agent responses in verified factual content via Retrieval-Augmented Generation (RAG). The KB must support multiple document formats, automatic processing, and live updates without system downtime.

Detailed Requirements
  • Ingestion formats: PDF, Word (.docx), Excel (.xlsx), plain text, web URLs (scrape + chunk), CSV
  • Processing pipeline: automatic chunking (512 tokens, 50-token overlap), metadata extraction, embedding generation
  • Embedding model: multilingual with Indic language support — multilingual-e5-large or text-embedding-3-large with multilingual fine-tuning
  • Vector store: Pinecone or Weaviate with per-tenant namespace isolation
  • Retrieval: top-k=5 chunks retrieved by cosine similarity, reranked by cross-encoder (Cohere Rerank or BGE Reranker) for final relevance ordering
  • Update latency: new documents searchable within 1 hour of upload; no system restart required
  • KB versioning: every response metadata includes the KB version tag and chunk IDs used (audit trail)
  • Access control: strict tenant isolation — Tenant A's KB is inaccessible to Tenant B (enforced at namespace + API key level)
  • Admin review workflow: uploaded documents go through admin-approval step before becoming active in production retrieval

F11 Self-Learning Module P1 High

The system shall continuously improve agent quality by collecting call outcome signals, curating high-quality training data from resolved calls, and running a regular fine-tuning pipeline on both the LLM and intent classifier. All model updates must pass automated regression gates before deployment.

Online Learning Signals (Real-Time)
  • Call resolution signal: detected from call disposition tag (resolved / escalated / abandoned) or explicit user confirmation ("Yes, that answered my question")
  • Escalation signal: call transfer to human agent is a negative quality signal logged against the LLM response that preceded it
  • CSAT signal: post-call SMS/IVR rating (1–5 stars) linked to call session and specific agent turns
  • Call duration signal: anomalously long calls (2x AHT) flagged for human review as potential confusion indicators
Offline Learning Pipeline
  • Weekly: curate high-quality resolved calls (CSAT ≥4, no escalation) as fine-tuning training data
  • Bi-weekly: LoRA/QLoRA fine-tune LLM on curated domain call transcript dataset; evaluate against baseline before deployment
  • Monthly: retrain intent classifier with new examples from live call distribution
  • Human annotation: sample 2% of all calls for human labeling in Label Studio; labeled data feeds into training pipeline
RLHF & A/B Testing
  • Reward model trained on human preference labels (which response was better?)
  • DPO (Direct Preference Optimization) or PPO fine-tuning using reward model
  • New model versions deployed to 10% traffic canary for 48 hours; promoted to 100% only after statistical evaluation
  • A/B test framework measures CSAT, escalation rate, resolution rate, and AHT as primary evaluation metrics

F13 Call Analytics & Reporting P1 High

The system shall provide a real-time and historical analytics dashboard covering both technical AI performance metrics and business outcome metrics. Full call transcripts shall be searchable, and automated reports shall be delivered on a configurable schedule.

Technical Metrics (Real-Time)
  • ASR WER per language and per time period
  • LLM response latency: P50, P95, P99
  • TTS first audio chunk latency
  • End-to-end call latency (ASR+LLM+TTS combined)
  • Concurrent active call count and queue depth
Business Metrics (Dashboard)
  • Call volume: total, by intent, by language, by time of day, by date range
  • Containment rate: percentage of calls fully resolved by AI without human escalation
  • Resolution rate: percentage of calls where the user query was successfully answered
  • Escalation rate: percentage of calls transferred to a human agent
  • Average Handle Time (AHT)
  • First Call Resolution (FCR)
  • CSAT score (where post-call survey collected)
  • Cost per call (computed from usage metering)
Call Quality & Audit
  • Full call transcript stored and searchable by keyword within 5 minutes of call end
  • Intent classification log, RAG retrieval chunk IDs, and confidence scores per turn
  • Timestamps for each pipeline stage (ASR start/end, LLM start/end, TTS start/end)
  • Optional call recording per tenant (compliance-configured; stored in encrypted S3)
  • Automated daily/weekly/monthly email reports; custom date-range CSV export

F15 Security, Compliance & Data Privacy P0 Critical

The system shall enforce enterprise-grade security controls and comply with Indian and international data protection regulations including GDPR, India's DPDP Act 2023, RBI data localization guidelines, and IRDAI call recording requirements. All PII must be masked in stored transcripts.

Data Security
  • All audio streams and API traffic encrypted with TLS 1.3 — no plaintext transmission
  • PII detection and masking in stored transcripts: Aadhaar numbers, bank account numbers, phone numbers, PAN, OTP values
  • Encryption at rest: AES-256 for all databases, S3 buckets, and vector stores
  • No audio files retained by default; configurable per tenant for compliance needs (IRDAI recording)
  • All secrets (API keys, DB credentials) managed via AWS Secrets Manager or HashiCorp Vault
Regulatory Compliance
  • GDPR: right to erasure (data deleted within 30 days of request), data portability (export in machine-readable format)
  • India DPDP Act 2023: data fiduciary obligations, explicit consent management, purpose limitation
  • RBI guidelines: payment-related data stored exclusively in AWS ap-south-1 (Mumbai) — no cross-border transfer
  • IRDAI: insurance call recordings retained per IRDAI-specified duration; accessible only to compliance team
  • SOC2 Type II: audit-ready logging of all data access events; annual SOC2 audit scheduled
Authentication & Access Control
  • Multi-factor authentication (MFA) mandatory for all admin portal users
  • Role-based access control (RBAC): Admin, Viewer, MLOps Engineer, Developer, Compliance Officer
  • API authentication: API key + short-lived JWT token (1-hour expiry)
  • IP whitelist option for enterprise tenants
  • Failed login lockout: 5 attempts → 15-minute lockout; 10 attempts → account suspension notification

F18 Evaluation & Quality Benchmarking P1 High

The system shall maintain a continuous, automated evaluation framework that benchmarks all AI components (ASR, LLM, TTS) against defined quality thresholds on a regular cadence. Regression gates shall block any model deployment that degrades existing quality metrics beyond acceptable tolerance.

ASR Evaluation
  • WER measured per language on curated held-out test sets (never used in training)
  • Real-time WER monitoring: alert triggered if WER degrades >20% relative vs rolling 7-day baseline
  • Benchmark datasets: AI4Bharat IndicSUPERB (Indic languages), Mozilla Common Voice (multilingual), internal telephony test set
LLM Evaluation
  • Intent classification accuracy on domain-specific test set (BFSI + Automotive)
  • Factual accuracy: hallucination detection rate on adversarial prompts
  • Response quality scoring: relevance, conciseness, grammatical correctness (automated via GPT-4 judge + periodic human review)
TTS Evaluation
  • MOS score tracked quarterly via listener panel (20 native speakers per language)
  • First-chunk latency monitored continuously; alert on degradation >50ms vs baseline
End-to-End Evaluation
  • Weekly call simulation: 50 synthetic calls per vertical run against production agent; resolution rate and containment rate measured
  • Regression gate: any metric degrading >2% vs baseline blocks deployment automatically
  • Human evaluation: 100 calls per week sampled for human scoring by trained annotators within 72 hours

§R

Risks & Mitigations

1
ASR accuracy poor on noisy telephony audio
Likelihood: High  |  Impact: High
Use telephony-optimized ASR models (Deepgram Nova-2 or Soniox); implement RNNoise pre-processing for 8kHz noise suppression; collect real telephony call data for regular domain fine-tuning; maintain WER monitoring alerts for early detection of degradation.
2
LLM hallucination in high-stakes BFSI domains
Likelihood: Medium  |  Impact: Critical
Enforce strict RAG-only responses for all factual queries; implement post-generation guardrail layer with hallucination detection; apply confidence threshold — if <0.70, route to human specialist; QA team runs weekly adversarial hallucination testing; zero-tolerance policy: any confirmed hallucination triggers immediate investigation.
3
Latency spikes under peak load
Likelihood: Medium  |  Impact: High
Load test to 2x expected peak before each major release; implement auto-scaling with 5-minute warm-up time; circuit breakers on all inter-service calls; local/lightweight TTS fallback for latency recovery (lower MOS but faster); LLM response caching for frequent queries (intent+entity hash).
4
Data breach or PII exposure
Likelihood: Low  |  Impact: Critical
PII masking applied at transcript write time (not post-hoc); no audio retention by default; all secrets in Vault/Secrets Manager; SOC2 Type II certification; quarterly internal security audits; bug bounty program; incident response plan with 72-hour notification SLA per DPDP Act.
5
Regulatory non-compliance (RBI / IRDAI / DPDP)
Likelihood: Medium  |  Impact: Critical
Legal review of all compliance requirements before MVP launch; RBI data localization enforced via AWS Config Rules; IRDAI call recording compliance built into platform configuration; in-house compliance officer engaged from day 1; external legal counsel for regulatory interpretation; compliance review scheduled quarterly.

Automatic Speech Recognition — converts spoken audio to text

Average Handle Time — mean duration of a call including wrap-up

Customer Satisfaction Score — post-call rating collected from callers

Digital Personal Data Protection Act (India, 2023)

Direct Preference Optimization — LLM alignment technique
FCR
First Call Resolution — query resolved without callback or escalation
G.711
ITU standard PCM audio codec used in telephony at 8kHz
LLM
Large Language Model — generative AI model for text understanding and generation
LoRA
Low-Rank Adaptation — parameter-efficient fine-tuning method for LLMs
MOS
Mean Opinion Score — perceptual audio quality rating (1–5 scale)
PPO
Proximal Policy Optimization — reinforcement learning algorithm used in RLHF
RAG
Retrieval-Augmented Generation — grounding LLM responses in retrieved documents
RBAC
Role-Based Access Control — permissions tied to user roles
RLHF
Reinforcement Learning from Human Feedback — alignment training using human preferences
RPO
Recovery Point Objective — maximum acceptable data loss window
RTO
Recovery Time Objective — maximum acceptable downtime window after failure
RSA
Roadside Assistance — emergency vehicle support service in Automotive vertical
SSML
Speech Synthesis Markup Language — XML-based TTS control format
STS
Speech-to-Speech — end-to-end voice pipeline: ASR → LLM → TTS
TTS
Text-to-Speech — converts text to synthesized spoken audio
VAD
Voice Activity Detection — identifies speech vs silence in audio stream
VAPT
Vulnerability Assessment and Penetration Testing
WER
Word Error Rate — primary ASR accuracy metric (lower is better)
QLoRA
Quantized Low-Rank Adaptation — memory-efficient LLM fine-tuning method
