System Architecture Overview

Architecture Pattern

Vaani AI uses a Cascaded STS (Speech-to-Speech) microservices architecture deployed on Kubernetes. Each processing stage is an independently scalable service communicating over gRPC internally, with REST management APIs and WebSocket real-time audio streams.

System Components

  1. Telephony Gateway Service — handles Twilio/Exotel webhooks, SIP trunking, WebRTC bridge
  2. Audio Processing Service — VAD (Voice Activity Detection), noise reduction, audio format/sample-rate conversion
  3. ASR Service — streaming speech recognition, language ID detection, partial transcript emission
  4. NLU Service — intent classification, named entity recognition, slot filling state machine
  5. Dialogue Manager — call script FSM execution, context orchestration, tool call routing
  6. LLM Service — prompt construction, RAG injection, response generation, streaming, guardrails
  7. TTS Service — text-to-speech with sentence-level streaming, voice/language selection per tenant
  8. Self-Learning Service — signal collection, dataset curation, training pipeline orchestration
  9. Analytics Service — metrics computation, business KPIs, dashboard data APIs
  10. Admin API — tenant management, configuration, knowledge base (KB) CRUD
  11. Notification Service — post-call webhooks, alert dispatch, SMS/email summaries

Cascaded STS Pipeline Design

The pipeline targets <600ms end-to-end latency from end-of-user-speech to first audio byte delivered to caller. Each stage below includes its latency budget and implementation notes.

Stage 1 Call Connect Budget: 0ms (sync)
  • Twilio/Exotel fires POST /webhook/inbound to Telephony Gateway
  • Gateway resolves tenant from the dialed number (To field) — PostgreSQL lookup, Redis-cached after first hit
  • Session created in Redis: {session_id, tenant_id, caller_id: hash(From), start_time, state: "greeting"}
  • Caller profile loaded from PostgreSQL (name, preferred language, prior interaction count)
  • WebSocket stream negotiated with Twilio Media Streams for bidirectional audio
Stage 2 VAD & Audio Streaming Budget: 0–100ms
  • Audio arrives as 20ms µ-law 8kHz frames via WebSocket from Twilio
  • VAD implemented with Silero-VAD (ONNX, CPU, ~2ms per frame): classifies each frame as speech or silence
  • Speech frames buffered; silence frames discarded to reduce upstream bandwidth
  • Audio resampled from 8kHz → 16kHz via librosa or soxr if the target ASR model requires 16kHz
  • End-of-utterance detected via VAD silence threshold (300ms sustained silence → speech segment complete)
Stage 3 ASR Streaming Budget: 100–300ms
  • Speech frames streamed to ASR Service over gRPC bidirectional stream
  • ASR emits partial transcripts every 100–200ms for speculative early intent detection
  • Primary: Soniox — token-level language detection enables native code-switch (Hindi+English in same utterance)
  • Detected language stored in Redis session for TTS voice selection
  • Final transcript returned ~300–500ms after end-of-utterance detection
Stage 4 NLU Processing Budget: 300–400ms
  • Final transcript sent to NLU Service via gRPC unary call
  • Intent classifier: fine-tuned DeBERTa-v3-base or LLM-based classifier (few-shot, per-tenant)
  • NER extracts entities: account numbers, dates, currency amounts, customer names, vehicle chassis numbers
  • Slot filling state check: if required slots for intent are missing → Dialogue Manager generates follow-up clarification question
  • Language detection confirmed: {language: "hi-IN", code_switch: true}
Stage 5 Context Assembly Budget: 400–420ms
  • Dialogue Manager assembles the LLM prompt from six sources:
  • 1. System prompt — tenant-specific, domain-specific (BFSI / Automotive), stored in PostgreSQL
  • 2. Script state — current FSM node, pending tasks, slot fill requirements
  • 3. Conversation history — last 20 turns pulled from Redis session
  • 4. Caller profile — name, language preference, prior intent history (from PostgreSQL)
  • 5. RAG context — top-5 KB chunks from Pinecone, retrieval runs in parallel with NLU (Stage 4) to hide latency
  • 6. Current turn — final transcript + detected intent + extracted entities
Stage 6 LLM Response Generation (Streaming) Budget: 420–600ms (first sentence)
  • LLM API called with fully assembled context; streaming mode enabled
  • Token stream decoded; first sentence boundary (period or newline) triggers immediate TTS dispatch
  • Remaining response streamed to TTS in sentence-sized chunks for continuous audio flow
  • Guardrail layer runs in parallel as a classifier: checks output for hallucinations, off-domain content, PII exposure, banned topics
  • If guardrail fires: response suppressed, safe fallback injected ("Let me connect you to a specialist")
Stage 7 TTS Streaming First audio at ~500–600ms
  • First sentence text sent to TTS Service (ElevenLabs Flash v2.5); first audio chunk returned in ~75ms
  • Audio immediately forwarded to caller via WebSocket
  • Subsequent sentence chunks queued and streamed as previous audio finishes playing — continuous, low-jitter output
  • Voice selection: per-tenant config (e.g., Hindi female voice ID for BFSI tenant, Tamil male for Automotive)
Stage 8 Post-Response (Async) Non-blocking / background
  • Full turn logged to MongoDB: {session_id, turn_id, transcript, intent, entities, response, latency_breakdown}
  • Call state updated in Redis; resolved intents added to resolved_intents[]
  • Analytics events published to Kafka topic call.turn.completed
  • If tool call needed (balance lookup, service appointment): executed in parallel via Celery task; result injected into next turn context
  • PII masking applied before log persistence (account numbers, phone numbers tokenized)

End-to-End Data Flow

Inbound Call Flow — Step by Step

  1. Caller dials tenant's phone number (Twilio / Exotel)
  2. Telephony provider POSTs webhook to Telephony Gateway: {CallSid, From, To, Direction}
  3. Gateway resolves tenant from the To number → loads tenant config from PostgreSQL → Redis-cached after first hit
  4. Gateway creates Redis session: {session_id: uuid4(), tenant_id, caller_id: sha256(From), start_time, language: null, state: "greeting"}
  5. Gateway instructs Twilio to open WebSocket Media Stream; bidirectional audio link established
  6. Greeting dispatched immediately to TTS (no ASR needed): "Welcome to [TenantName]. How may I help you today?"
  7. TTS audio chunk returned in ~75ms → streamed to caller via WebSocket — caller hears greeting within ~100ms of call answer
  8. Caller speaks → 20ms µ-law audio frames arrive at Gateway via WebSocket
  9. VAD (Silero-VAD) classifies frames; speech segments buffered and streamed to ASR Service over gRPC
  10. ASR Service (Soniox) emits partials: ["नमस्ते", "नमस्ते मेरा", "नमस्ते मेरा account"]
  11. Language detected as hi-IN with code-switch flag; stored in Redis session
  12. End-of-utterance detected → final transcript: "नमस्ते मेरा account balance बताइए"
  13. NLU Service: {intent: "check_balance", entities: {}, language: "hi-IN", code_switch: true}
  14. Dialogue Manager checks script FSM: state "greeting" → transition requires auth before balance → new state "auth"
  15. PARALLEL: RAG retrieval begins on query "account balance check process" → Pinecone returns top-10 → cross-encoder reranks → top-5 selected
  16. Context assembled: system prompt + script state(auth) + last 20 turns + caller profile + RAG chunks + current intent
  17. LLM (Claude 3.5 Sonnet) called with streaming; generates: "आपका balance जानने के लिए, कृपया अपना account number बताएं"
  18. Guardrail classifier: response passes — on-domain, no hallucination, no PII exposure
  19. First sentence dispatched to TTS → ElevenLabs Hindi voice → audio chunk in 75ms → to caller
  20. Turn logged async to MongoDB; analytics event published to Kafka: {session_id, turn_id, language, intent, latency_asr_ms, latency_llm_ms, latency_tts_ms}
  21. Next turn: Caller provides account number → NLU extracts entity → auth service verifies → balance fetched from core banking API → response generated with actual balance
  22. Call end: session closed, call summary written to PostgreSQL, learning signals processed by Self-Learning Service
Latency target: From end-of-user-speech to first audio byte delivered to caller — target <600ms P95. Steps 9–19 above occur within this window. RAG retrieval (Step 15) runs in parallel with NLU (Step 13) to avoid adding to the critical path.

Weekly Curated Dataset Generation

-- PostgreSQL: select high-quality sessions for training SELECT s.session_id, s.transcript_id FROM sessions s WHERE s.resolution_status = 'RESOLVED' AND s.csat_score >= 4 AND s.escalation_count = 0 AND s.created_at >= NOW() - INTERVAL '7 days'; -- Transcripts fetched from MongoDB, formatted as instruction pairs: -- [{"role": "system", "content": system_prompt}, -- {"role": "user", "content": caller_utterance}, -- {"role": "assistant", "content": agent_response}]

Human Review Pipeline

  • 10% of auto-selected samples reviewed in Label Studio by domain QA annotators
  • Annotators flag: incorrect intent labels, hallucinated responses, poor code-switch handling
  • Final curated dataset versioned and stored in S3: s3://vaani-ai-training/datasets/v{date}/

Bi-Weekly Fine-Tuning Pipeline

# HuggingFace PEFT — LoRA fine-tuning config from peft import LoraConfig, get_peft_model lora_config = LoraConfig( r=16, lora_alpha=32, target_modules=["q_proj", "v_proj"], lora_dropout=0.05, bias="none", task_type="CAUSAL_LM" ) # Training: 3 epochs, lr=2e-4, batch_size=16 on 1x A100 # Estimated time: 2–4 hours for 10K examples

Evaluation Gate & Deployment

  • New model evaluated on held-out test set (20% of dataset, never used in training)
  • Must meet thresholds: intent accuracy ≥ (baseline − 0.5%), resolution rate ≥ (baseline − 1%)
  • If passes gate → canary deployment at 5% traffic → monitor 48 hours → progressive rollout to 100%
  • MLflow Model Registry tracks all experiments, evaluation metrics, and model artifacts with full lineage

RAG (Retrieval Augmented Generation) Design

Ingestion Pipeline

  1. Document uploaded via Admin API: PDF, Word (.docx), Excel, or URL
  2. Document Processor: PyMuPDF for PDF text extraction; python-docx for Word; pandas for Excel; trafilatura for URLs
  3. Chunker: LangChain RecursiveCharacterTextSplitter, chunk_size=512, chunk_overlap=50 tokens; respects paragraph and sentence boundaries
  4. Embeddings:
    • English-heavy content: text-embedding-3-large (OpenAI, 3072 dims)
    • Indic / multilingual content: multilingual-e5-large (Hugging Face, 1024 dims, trained on Indic text)
  5. Upsert to Pinecone: namespace = {tenant_id}_{domain}; metadata = {doc_id, title, section, page, updated_at, language}
  6. KB version record updated in PostgreSQL; cache invalidation signal published to Redis

Retrieval at Query Time

# Runs in parallel with NLU (Stage 4) to hide latency async def retrieve_rag_context(query: str, tenant_id: str, language: str): embedding = await embed(query, model=select_embed_model(language)) # Pinecone ANN search — returns top 10 candidates candidates = pinecone_index.query( vector=embedding, top_k=10, namespace=f"{tenant_id}_bfsi", include_metadata=True ) # Cross-encoder reranking — selects best 5 reranked = cross_encoder.predict( [(query, c["metadata"]["text"]) for c in candidates] ) top5 = sorted(zip(candidates, reranked), key=lambda x: -x[1])[:5] # Citation tracking: every chunk has source doc ID return [{ "text": c["metadata"]["text"], "doc_id": c["metadata"]["doc_id"] } for c, _ in top5]

Citation & Audit Trail

  • Every LLM response is tagged with the source doc IDs used in the context (used_doc_ids: ["doc_123", "doc_456"])
  • RAG retrieval log stored per call turn in MongoDB for compliance auditing
  • Cross-encoder model: cross-encoder/ms-marco-MiniLM-L-6-v2 — fast (~5ms on CPU), accurate relevance reranking

Security Architecture

PII Handling

  • Phone number hashing: Raw phone numbers never stored; SHA-256 hash used as caller_id throughout the system
  • In-flight masking: Before logging to MongoDB, NER-based PII masker redacts account numbers, Aadhaar, PAN, credit card numbers using regex + ML classifier
  • PII masking levels: HIGH (financial data fully tokenized), MEDIUM (partial masking), LOW (names retained if consented)
  • Data residency: All AWS resources deployed in ap-south-1 (Mumbai) for Indian tenants; EU tenants on eu-west-1

Authentication & Authorization

  • Admin API: JWT-based auth (RS256); roles: super_admin, tenant_admin, analyst
  • Webhook validation: Twilio webhook signatures verified using HMAC-SHA1 on every inbound call event
  • Internal services: mTLS enforced between all Kubernetes pods via Istio service mesh
  • API keys: Third-party API keys (Soniox, ElevenLabs, OpenAI) stored in AWS Secrets Manager; rotated every 90 days

Encryption

  • In transit: TLS 1.3 on all external endpoints; gRPC uses TLS by default; WebSocket over WSS (TLS)
  • At rest: PostgreSQL, MongoDB (DocumentDB), Redis, S3 — all encrypted with AWS KMS (AES-256); per-tenant KMS key option for enterprise clients
  • Audio recordings: If call recording enabled (BFSI compliance), audio stored in S3 with SSE-KMS; access gated by IAM role with CloudTrail audit

Compliance Controls

  • India DPDPA 2023: Consent management for cross-call memory (consent_cross_call boolean per caller); data deletion API (DELETE /admin/callers/{id})
  • TRAI DND: Exotel handles DND compliance for India outbound calls; inbound calls exempt
  • SOC 2 Type II: Logging of all admin actions in immutable CloudTrail; vulnerability scanning via AWS Inspector; penetration test annually
  • RBI IT Framework: For BFSI tenants — data localization (Mumbai region), audit logs retained 5 years, encryption key management documented

Guardrails (LLM Output Safety)

  • Domain classifier: Fine-tuned binary classifier runs on every LLM output; blocks off-domain responses (e.g., medical/legal advice from a BFSI agent)
  • Hallucination detection: Response cross-checked against provided RAG chunks; if factual claim cannot be grounded in KB — suppressed
  • Banned topic list: Per-tenant configurable blocklist (e.g., competitor names, regulatory topics requiring human handling)
  • Fallback policy: On guardrail trigger → "Let me connect you to a specialist" → human escalation pathway initiated

  • Log levels: ERROR / WARN / INFO / DEBUG; production runs at INFO; debug enabled per-session via feature flag in Redis

Model Training & Fine-Tuning

Domain Adaptation — BFSI

  • Custom ASR vocabulary: NACH, SWIFT, CIBIL, Aadhaar, FD (Fixed Deposit), RD (Recurring Deposit), EMI, NPA, SARFAESI, NACH mandate, NEFT, RTGS, UPI, IMPS, LTV, CAGR
  • Pronunciation lexicon: ₹ → "rupees"; HDFC → "H-D-F-C"; ICICI → "I-C-I-C-I"; SBI → "S-B-I"; ATM → "A-T-M"; OTP → "O-T-P"
  • LLM instruction fine-tuning: 5,000 curated BFSI Q&A pairs (loan queries, account services, credit card, FD/RD, complaint resolution)
  • Evaluation: BFSI domain Q&A accuracy on 500 held-out questions; target ≥ 92% correct with no hallucinated figures

Domain Adaptation — Automotive

  • Custom ASR vocabulary: OBD2, BS6, TPMS, VIN, chassis number, RSA (Roadside Assistance), CNG, EV charging, AMT transmission, ARAI, VAHAN
  • LLM fine-tuning: 3,000 Automotive service + sales Q&A pairs covering Maruti, Tata, Hyundai, Mahindra, Honda, Kia model-specific terminology
  • Use case coverage: Service appointment booking, service cost estimation, warranty queries, roadside assistance dispatch, EV charging station lookup

CI/CD Pipeline

  • Source control: GitHub (monorepo, one directory per service)
  • CI: GitHub Actions — on PR: lint (ruff, mypy), unit tests (pytest), Docker build, security scan (Trivy)
  • CD: ArgoCD (GitOps) — Helm chart updates in infra/ repo trigger rolling deployments to EKS
  • Container registry: Amazon ECR; images tagged with git SHA + semantic version
  • Secrets: AWS Secrets Manager → Kubernetes External Secrets Operator → mounted as env vars; never in git
  • Feature flags: Redis-backed per-tenant flags for gradual rollout of new ASR/LLM/TTS providers

Disaster Recovery

  • RTO: < 30 minutes; RPO: < 5 minutes
  • PostgreSQL: automated daily snapshots + continuous WAL archival to S3; cross-region replication to ap-southeast-1 for DR
  • Redis: AOF persistence enabled; ElastiCache automated backups daily
  • EKS: Infrastructure as Code via Terraform; cluster can be recreated in < 20 minutes from scratch
